# 딸깍 콘테스트 (애니멀 리그)

최종 기획안 v1.0

## 1. 프로젝트 개요

### 1.1 프로젝트명

- 딸깍 콘테스트 (애니멀 리그)

### 1.2 한 줄 설명

- 공부 타이머 앱에 교수 캐릭터 공략형 미연시 구조를 결합한 모바일 웹 기반 게이미피케이션 학습 서비스

### 1.3 핵심 컨셉

- 유저는 공부를 통해 교수 캐릭터의 호감도를 올린다.
- 공부 종료 후 확률적으로 시나리오 이벤트가 발생한다.
- 유저의 목표는 대사를 보기 위해 공부를 지속하는 것이다.
- 최종적으로 시험 점수를 입력해 엔딩 대상 교수를 선택한다.

## 2. 서비스 목표

### 2.1 제품 목표

- 공부 지속 시간을 늘린다.
- 공부 행동에 즉각적인 감정적 보상을 부여한다.
- 반복 접속과 재사용을 유도한다.

### 2.2 사용자 경험 목표

- "공부하면 캐릭터 반응을 볼 수 있다."
- "공부를 끊거나 방치하면 캐릭터가 반응한다."
- "교수별 다른 대사와 반응을 수집하고 싶다."

## 3. 플랫폼 및 실행 환경

### 3.1 플랫폼

- 모바일 웹 실행 기준

### 3.2 사용 시나리오

1. 사용자는 모바일 브라우저로 접속한다.
2. 로그인 후 교수 캐릭터를 등록한다.
3. 공부 시작 버튼을 눌러 세션을 시작한다.
4. 공부 종료 버튼을 눌러 세션을 종료한다.
5. 종료 결과에 따라 호감도, 대사, 이벤트가 반영된다.

### 3.3 전제

- 앱이 아니라 웹 퍼블리시 기준으로 설계한다.
- 모바일 환경에서의 탭 전환, 화면 비활성화, 백그라운드 전환을 고려한다.

## 4. 핵심 루프

### 4.1 메인 플레이 루프

1. 학교 접속
2. 교수 등록 또는 선택
3. 공부 시작
4. 공부 진행
5. 공부 종료
6. 대사 출력
7. 확률적으로 시나리오 이벤트 발생
8. 선택지 응답
9. 호감도 변화
10. 다음 공부 세션 진입

### 4.2 재미의 중심

- 이 서비스의 핵심 보상은 대사와 반응이다.
- 유저는 스탯이나 경쟁보다 캐릭터 반응을 보기 위해 공부한다.
- 따라서 호감도, 상황 대사, 시나리오 이벤트가 핵심 콘텐츠다.

## 5. 계정 및 기본 정보

### 5.1 계정 정보

- 이름
- ID
- 비밀번호
- 시험 종료일

### 5.2 시험 종료일의 역할

- 현실 시간 기준으로 최종 결과 시점을 잡는 핵심 변수다.
- 시험 종료일 도달 후 결과 입력 및 엔딩 분기가 가능하다.

## 6. 교수 등록 시스템

### 6.1 입력값

- 교수 이름
- 교수 성별
- 교수 사진
- 교수 성격

성별 값:

- `male`
- `female`

### 6.2 성격 설정

교수 캐릭터 생성은 교수 사진, 성별, 성격을 기준으로 진행한다.

예시 성격 타입:

- `gentle`: 다정한 교수님
- `tsundere`: 나한테만 츤데레 냉미남 교수님
- `english_mix`: 영어 섞어쓰는 유쾌한 교수님
- `shy`: 수줍은 교수님

### 6.3 캐릭터 에셋 생성

- 사용자가 교수 사진을 입력하면, 해당 사진을 기준으로 `nanobanana`를 사용해 여러 표정과 포즈의 전신 2D 캐릭터 이미지 세트를 생성한다.
- 생성된 캐릭터 에셋은 메인 화면, 이벤트 팝업, 대사 출력, 결과 화면 등 게임 내 교수 캐릭터 표현에 사용한다.
- 에셋은 사전에 정의된 표정/포즈 슬롯에 맞춰 관리한다.

### 6.4 사진 예외 처리

- 교수 사진이 없을 경우 기본 이미지를 사용한다.
- 교수 사진 기반 생성이 실패할 경우에도 게임 진행이 막히지 않도록 기본 캐릭터 에셋 세트를 사용한다.

### 6.5 설계 원칙

- MVP에서는 자동 성격 추론 등은 제외한다.
- MVP에서는 교수 사진 기반 캐릭터 에셋 생성만 포함하고, 텍스트만으로 신규 캐릭터를 임의 생성하는 기능은 제외한다.
- 캐릭터 반응은 사전에 정의된 성격 템플릿을 기준으로 출력한다.

## 7. 공부 세션 시스템

### 7.1 세션 시작

- 유저가 공부 시작 버튼을 누르면 세션이 시작된다.
- 시작 시점부터 시간을 측정한다.

### 7.2 세션 종료

- 유저가 공부 종료 버튼을 누르면 세션이 종료된다.
- 종료 시 공부 시간이 반영된다.
- 종료 후 대사가 출력된다.
- 30% 확률로 시나리오 이벤트가 발생한다.

### 7.3 세션 구조

- 공부 세션 길이는 유저 자유다.
- 고정 50분, 포모도로 구조는 아니다.
- 사용자가 원하는 시점에 시작과 종료가 가능하다.

## 8. 호감도 시스템

### 8.1 기본 구조

- 교수별 호감도 값은 각각 유지된다.
- 공부 세션으로 얻는 보상은 등록된 모든 교수에게 동일하게 반영된다.
- 이벤트 선택과 세션 대상 교수 기준 패널티는 해당 교수에게만 반영된다.
- 호감도 최대치는 100이다.

### 8.2 호감도 상승

공부 세션 보상:

- 10분당 +2
- 한 세션 최대 +10
- 계산된 보상은 등록된 모든 교수에게 동일하게 가산된다.

예시:

- 10분 공부: +2
- 30분 공부: +6
- 50분 이상 공부: 최대 +10

### 8.3 호감도 감소

방치 패널티:

- 방치 확정 시 아래 규칙을 적용한다.

```text
penalty = max(현재 호감도 // 10, 1)
```

중도 포기 패널티:

공부가 충분히 누적되기 전에 종료한 경우 아래 규칙을 적용한다.

```text
effectiveStudySeconds < 1200 이면 penalty = 2
```

예시:

- 유효 공부 시간 19분 59초: -2
- 유효 공부 시간 20분 이상: 0
- 호감도 20~29: -2
- 호감도 50~59: -5
- 호감도 100: -10

### 8.4 설계 의도

- 초반에는 부담 없이 접근 가능하다.
- 후반에는 관계를 유지하는 긴장감이 발생한다.
- 공부를 꾸준히 이어가는 동기를 만든다.

## 9. 방치 감지 시스템

### 9.1 목적

모바일 웹 환경에서 유저가 공부 중 실제로 이탈했는지 감지하고, 방치 시 패널티와 반응을 발생시킨다.

### 9.2 기준 기술

- Page Visibility API 사용
- 모바일 브라우저 실행 기준

### 9.3 판정 방식

- 공부 중일 때 프론트엔드에서 현재 시간을 주기적으로 기록한다.
- 모바일 브라우저가 비활성화되거나 백그라운드로 전환되었을 때 `visibilitychange` 이벤트를 감지한다.

### 9.4 방치 판정 규칙

- 공부 중 `document.visibilityState`가 `hidden`으로 전환되면 이탈 상태 후보로 기록한다.
- 이후 다시 `visible` 상태가 되었을 때 이탈 시간을 계산한다.
- 숨김 상태가 2분 이상 지속되면 방치로 확정한다.
- 방치 확정 시 해당 세션에 대해 패널티 `-2`를 적용한다.

### 9.5 처리 원칙

- 방치 패널티는 한 번의 이탈 구간당 1회만 적용한다.
- 방치 중인 시간은 공부 시간에 포함하지 않는다.
- 복귀 시 복귀 대사 또는 경고 대사를 출력할 수 있다.

### 9.6 모바일 기준 장점

- 별도 입력 이벤트 추적 없이 구현 가능하다.
- 탭 전환, 앱 전환, 화면 비활성화 등 모바일 사용 패턴에 대응 가능하다.
- MVP 수준에서 구현 복잡도를 낮출 수 있다.

## 10. 이벤트 시스템

### 10.1 대사 발생 시점

다음 상황에서 대사가 출력된다.

1. 최초 접속 시
2. 여러 인터랙션 시
3. 공부 종료 시
4. 방치 후 복귀 시
5. 기타 상황별 시나리오 이벤트

### 10.2 최초 접속 대사

- 유저의 첫 접속 시 1회 출력한다.
- 캐릭터 소개 및 첫 인상 형성 역할을 한다.

### 10.3 인터랙션 대사

- 화면 내 특정 인터랙션 발생 시 출력한다.
- 예: 교수 터치, 버튼 액션, 화면 진입 등

### 10.4 공부 종료 대사

- 공부 종료 시 기본 반응 대사를 출력한다.

### 10.5 시나리오 이벤트

- 공부 종료 후 30% 확률로 발생한다.
- 짧은 대사 출력이 아니라 별도 선택지가 포함된 이벤트다.
- 이벤트는 호감도 구간 `0-19`, `20-49`, `50-84`, `85-100`을 기준으로 다른 풀에서 선택된다.
- 이벤트 원문은 구간 공통 파일과 교수 성격별 분기 파일을 조합하는 방식으로 관리한다.

## 11. 미연시 이벤트 시스템

### 11.1 발생 조건

- 공부 종료 시 30% 확률

### 11.2 구성

- 대사
- 선택지
- 결과 반응
- 호감도 변화

### 11.3 선택지 개수

- 고정하지 않는다.
- 이벤트에 따라 가변 구성한다.

### 11.4 정답 구조

- 정답 기준은 이미 정의되어 있다.
- 교수별 성격과 이벤트 맥락에 따라 정답 선택지가 다르다.

### 11.5 결과 처리

- 올바른 답변 선택 시 호감도가 상승한다.
- 잘못된 답변 선택 시 호감도가 감소하거나 무반응 처리된다.

### 11.6 설계 원칙

- 랜덤 찍기보다 캐릭터 이해를 통해 답을 유추하는 구조다.
- 교수 성격을 학습할수록 유리한 경험을 제공한다.

## 12. 상황별 대사 분류

### 12.1 기본 분류

1. 처음 접속했을 때
2. 당일 최초 접속
3. 학교 밖에서 돌아왔을 때
4. 공부 시작 버튼을 눌렀을 때
5. 오래 멈춰 있었을 때
6. 공부 재시작할 때
7. 공부 중 이탈했을 때
8. 응원 대사
9. 가장 많이 공부한 교수와 가장 적게 공부한 교수의 한마디
10. 최종 결과 대사

### 12.2 역할

- 반복 행동에 감정적 피드백을 부여한다.
- 캐릭터별 말투 차별화를 만든다.
- 학습 행동을 캐릭터 관계와 연결한다.

## 13. 최종 결과 및 엔딩

### 13.1 결과 진입 조건

- 시험 종료일 도달 후 유저가 최종 시험 점수를 입력한다.

### 13.2 엔딩 대상 선정

- 유저가 입력한 최종 시험 점수를 기준으로 결과를 본다.
- 결과 출력 시 특정 교수 중심 엔딩을 노출한다.

### 13.3 결과 대사

- 가장 호감도가 높은 교수의 한마디
- 가장 호감도가 낮은 교수의 한마디
- 성적 결과에 대한 교수별 반응 대사

### 13.4 호감도 구간 반영

최종 결과 대사는 최소 아래 구간을 반영한다.

- 호감도 85~100
- 호감도 50~84
- 호감도 20~49
- 호감도 0~19

### 13.5 설계 의도

- 엔딩은 단순 성적 공개가 아니라 공부 결과와 캐릭터 관계 결과를 함께 느끼게 하는 장치다.

## 15. 상태 정의

### 15.1 주요 상태

- `idle`: 기본 대기 상태
- `studying`: 공부 진행 중
- `hidden`: 모바일 웹 비활성 상태
- `returned`: 복귀 상태
- `event`: 시나리오 이벤트 진행 중
- `result`: 최종 결과 확인 상태

### 15.2 공부 상태 전이

- `idle` -> 공부 시작 -> `studying`
- `studying` -> visibility hidden -> `hidden`
- `hidden` -> 2분 이상 후 복귀 -> 방치 확정
- `hidden` -> 2분 미만 후 복귀 -> 정상 복귀
- `studying` -> 공부 종료 -> 이벤트 판정
- 이벤트 종료 -> `idle`

## 16. 콘텐츠 설계 원칙

### 16.1 캐릭터 차별화

- 교수별 말투, 태도, 반응 패턴이 달라야 한다.
- 같은 상황에서도 성격별 대사가 달라야 한다.

### 16.2 대사 우선 설계

이 서비스는 대사가 핵심 보상이다. 따라서 UI보다 먼저 대사 데이터 구조와 분류 체계를 설계해야 한다.

### 16.3 반복 피로 대응

- 대사 풀을 충분히 확보해야 한다.
- 인터랙션 대사, 공부 종료 대사, 방치 대사, 시나리오 이벤트 대사를 분리 관리해야 한다.

## 20. 최종 정리

이 서비스의 본질은 공부 도구가 아니라, 공부 행동을 감정 보상 루프로 바꾸는 모바일 웹형 캐릭터 인터랙션 서비스다.

핵심은 아래 세 가지다.

- 자유 공부 세션
- 모든 교수 공통 공부 보상과 교수별 개별 반응
- 대사를 보기 위한 공부 동기

그리고 모바일 환경에서는 Page Visibility API 기반 방치 감지를 표준으로 삼아 탭 이탈과 복귀를 중심으로 공부 흐름을 관리한다.
