# 작업 목록: [FEATURE NAME]

**입력**: `/specs/[FEATURE_ID]/`의 설계 문서
**선행 문서**: plan.md, spec.md, research.md, data-model.md, contracts/
**테스트**: 모든 사용자 스토리는 TDD에 따라 실패하는 테스트부터 시작해야 한다.
**구성 원칙**: 작업은 사용자 스토리 단위로 묶고 `frontend/`, `backend/`, `Docs/` 기준으로 나누어 FE와 BE가 숨은 의존성 없이 병렬 작업할 수 있게 한다.

## 형식: `[ID] [P?] [Story] 설명`

- **[P]**: 병렬 실행 가능 (서로 다른 파일, 미완료 의존성 없음)

- 모든 항목 설명에는 정확한 파일 경로를 포함한다.

## 경로 규칙

- **프론트엔드**: `frontend/src/`, `frontend/tests/`
- **백엔드**: `backend/src/main/`, `backend/src/test/`
- **문서**: `Docs/`
- **GitHub 워크플로**: `.github/`

## 의존성과 실행 순서

### 단계 의존성

- **준비 (1단계)**: 의존성 없음
- **기반 작업 (2단계)**: 준비 단계 완료 후 시작하며 계약을 고정한다.
- **사용자 스토리 (3단계 이후)**: 기반 작업 완료 후 시작한다.
- **마무리 단계**: 선택한 사용자 스토리 구현이 끝난 뒤 시작한다.

### 사용자 스토리 의존성

- 계약과 기반 작업이 고정되면 사용자 스토리는 병렬로 진행할 수 있다.
- 각 사용자 스토리는 반드시 독립적으로 테스트 가능하고 릴리스 가능해야 한다.

### 사용자 스토리 내부 규칙

- 테스트는 반드시 구현보다 먼저 작성되고 실패를 확인해야 한다.
- 공통 계약 작업이 끝나면 같은 스토리 안에서도 FE와 BE는 병렬로 작업할 수 있다.
- 문서 갱신은 구현과 함께 포함되어야 하며 나중으로 미루지 않는다.
- FE/BE 미해결 질문은 머지 전에 comment 문서에 기록해야 한다.

### 병렬 작업 기회

- [P]가 붙은 준비 단계 작업은 모두 병렬 실행할 수 있다.
- 기반 단계의 FE/BE 진입점 작업은 계약 초안 작성 이후 병렬로 진행할 수 있다.
- 테스트와 계약이 준비되면 같은 사용자 스토리의 FE/BE 구현 작업을 병렬로 진행할 수 있다.
- 인력과 의존성 경계가 허용되면 서로 다른 사용자 스토리도 병렬로 진행할 수 있다.

## 구현 전략

### MVP 우선 (사용자 스토리 1만 우선)

1. 준비 단계를 끝낸다.
2. 기반 작업을 끝낸다.
3. 사용자 스토리 1의 테스트를 먼저 작성한 뒤 구현한다.
4. FE/BE 연동과 문서 갱신을 검증한다.
5. 헌장이 허용한 대상 브랜치에서 배포 또는 데모를 진행한다.

### 점진적 전달

1. 계약을 먼저 고정한다.
2. 사용자 스토리 1을 테스트와 문서와 함께 전달한다.
3. 사용자 스토리 2를 테스트와 문서와 함께 전달한다.
4. 사용자 스토리 3을 테스트와 문서와 함께 전달한다.
5. 각 증분은 표준 PR 템플릿 기준으로 바로 머지 가능한 상태를 유지한다.

### 병렬 팀 전략

여러 개발자가 함께 작업할 때:

1. 공유 계약과 기반 작업을 함께 확정한다.
2. 계약 고정 후 사용자 스토리별 FE와 BE 소유권을 분리한다.
3. 차단 이슈는 `Docs/FE-comment-vX.Y.md`와 `Docs/BE-comment-vX.Y.md`에 기록한다.
4. 계약 검증과 CI를 통해 다시 통합한 뒤 머지한다.

## 메모

- [P] 작업은 서로 다른 파일을 다루며 미충족 의존성이 없다.
- 각 사용자 스토리는 독립적으로 완성하고 테스트할 수 있어야 한다.
- 계약 변경이 생기면 `Docs/`의 버전 문서를 함께 갱신해야 한다.
- 숨은 교차 팀 가정을 만들지 말고, 문서화하거나 제거한다.
