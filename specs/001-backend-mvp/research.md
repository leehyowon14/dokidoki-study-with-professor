# 조사 기록: 백엔드 MVP 기반

## 결정 1: 백엔드는 모듈형 모놀리스로 구현한다

- **결정**: `Backend/` 안에 단일 Spring Boot 애플리케이션을 두고, `auth`,
  `professor`, `study`, `dialogue`, `result`, `common` 모듈로 분리한다.
- **근거**: MVP 단계에서 마이크로서비스 분할은 운영 복잡도를 키우고, 현재 요구는
  강한 트랜잭션 일관성과 빠른 피드백이 더 중요하다. 동시에 도메인 패키지를 나누면
  2인 백엔드 병렬 작업도 가능하다.
- **검토한 대안**:
  - 레이어 기준 단일 패키지 구조: 변경 범위 추적과 모듈 소유권이 흐려져 병렬 작업성이 떨어짐
  - 마이크로서비스 분리: 배포/테스트/운영 비용이 MVP 범위에 비해 과함

## 결정 2: 인증은 Spring Security 기반 서버 세션으로 시작한다

- **결정**: 회원 가입과 로그인은 Spring Security + BCrypt + 서버 세션 기반으로
  시작하고, 로그인 응답은 사용자 정보와 활성 세션 상태를 반환한다.
- **근거**: 현재 계약 문서는 토큰 페이로드보다 로그인 결과와 활성 세션 상태를
  중심으로 정의되어 있다. 세션 기반 인증은 MVP에서 구현량을 줄이고 계정/세션 제약을
  단순하게 유지한다.
- **검토한 대안**:
  - JWT access/refresh token: FE/BE 분리에는 유리하지만 토큰 저장/폐기 전략까지 함께
    결정해야 해서 현재 MVP에는 과함
  - 외부 OAuth: 제품 요구에 없음

## 결정 3: PostgreSQL을 유일한 기준 저장소로 사용한다

- **결정**: 사용자, 교수, 호감도, 세션, 이벤트 결과, 최종 결과는 모두 PostgreSQL에
  저장한다. 교수 원본 사진과 생성된 캐릭터 에셋 파일은 스토리지 포트로 분리하고,
  DB에는 메타데이터와 URL만 저장한다.
- **근거**: 핵심 도메인은 관계가 뚜렷하고 트랜잭션 경계가 중요하다. PostgreSQL
  하나로 시작하면 도메인 무결성과 테스트 환경 구성이 단순하다.
- **검토한 대안**:
  - H2 기반 개발: 실제 DB와 동작 차이가 생겨 계약 정확도가 떨어질 수 있음
  - Redis 추가: 현재는 캐시 이득보다 운영 복잡도가 큼

## 결정 4: 교수 사진 기반 캐릭터 에셋 생성은 비동기 오케스트레이션으로 처리한다

- **결정**: 교수 등록 시 `sourcePhotoUrl`이 있으면 백엔드는 원본 사진 참조를 저장한 뒤
  `nanobanana` 기반 생성 작업을 비동기로 시작한다. 생성 대상은 사전 정의된
  `variantKey` 묶음의 전신 2D 캐릭터 에셋 세트이며, 원본 사진이 없거나 생성 실패 시
  기본 캐릭터 에셋 세트를 사용한다.
- **근거**: 교수 등록 요청에서 외부 이미지 생성이 동기식으로 수행되면 응답 지연과
  실패 전파가 커진다. 비동기 오케스트레이션은 등록 계약을 안정적으로 유지하면서도
  여러 표정/포즈 에셋 생성을 확장 가능하게 만든다.
- **검토한 대안**:
  - 등록 요청 안에서 동기 생성: 외부 의존성 실패가 가입 초기 흐름을 직접 깨뜨림
  - 대표 이미지 1장만 생성: 게임 내 다양한 감정/포즈 연출 요구를 충족하지 못함

## 결정 5: 공부 시간과 패널티 계산은 서버가 최종 책임진다

- **결정**: `StudySession`과 `StudySessionHiddenSegment`를 모두 저장하고, 세션 종료
  시 서버가 유효 공부 시간, 중도 포기 패널티, 방치 패널티, 최종 호감도 변화를 계산한다.
- **근거**: 제품 핵심 가치가 공부 종료 시점의 정산 정확성에 있다. 클라이언트가
  계산을 담당하면 조작 가능성과 FE/BE 불일치가 커진다.
- **검토한 대안**:
  - 클라이언트 계산 후 저장: 조작 가능성과 동기화 위험이 큼
  - 종료 시 숨김 구간 합산 없이 단순 타이머 기반 처리: 방치 규칙을 만족하지 못함

## 결정 6: 대사와 이벤트는 교수 우선, 성격 공통 대체 규칙을 사용한다

- **결정**: 대사와 이벤트 조회는 `professorId` 우선, 없으면 `personalityType`
  공통 콘텐츠를 대체 규칙으로 사용한다.
- **근거**: MVP에서 콘텐츠 볼륨 부족을 막으면서도, 특정 교수 차별화 요구를 유지할
  수 있다.
- **검토한 대안**:
  - 교수 전용 데이터만 사용: 콘텐츠 준비 비용이 급격히 증가
  - 성격 공통 데이터만 사용: 교수별 개성이 약해짐

## 결정 7: 계약 문서는 OpenAPI와 버전 문서를 함께 유지한다

- **결정**: 구현 전용 계약은 `specs/001-backend-mvp/contracts/backend-api.openapi.yaml`
  로 관리하고, 팀 공용 기준은 `Docs/api-spec-v0.1.md`를 유지한다. 대사와 시나리오
  구조 기준은 `Docs/dialogue-content-spec-v0.1.md`와 `Backend/Reference/`
  원문 문서를 함께 유지한다.
- **근거**: OpenAPI는 테스트와 서버 구현에 직접 쓰기 좋고, 버전 문서는 FE/BE
  소통과 PR 리뷰에서 읽기 쉽다. 시나리오 원문과 렌더 규칙은 별도 기준 문서가 있어야
  콘텐츠 변경이 API 계약과 분리되지 않는다.
- **검토한 대안**:
  - Docs Markdown만 유지: 자동화와 계약 테스트 활용성이 떨어짐
  - OpenAPI만 유지: 비개발 이해관계자와 FE 협업 시 가독성이 떨어질 수 있음

## 결정 8: 테스트는 계약 -> 통합 -> 단위 순으로 계층화한다

- **결정**: 엔드포인트 응답 형상은 계약 테스트, 주요 사용자 흐름은 통합 테스트,
  계산식과 순수 규칙은 단위 테스트로 분리한다.
- **근거**: TDD를 강제하면서도 실패 원인을 빠르게 좁히려면 테스트 층을 분리해야
  한다. 특히 비즈니스 계산식은 단위 테스트에서 빠르게 검증하고, 최종 계약은 MockMvc와
  Testcontainers로 보증하는 구성이 적절하다.
- **검토한 대안**:
  - 통합 테스트만 사용: 느리고 실패 원인 분석이 어렵다
  - 단위 테스트만 사용: HTTP 계약과 직렬화 문제를 잡지 못한다

## 결정 9: 백엔드 2인 협업은 도메인 축으로 분할한다

- **결정**: 개발자 A는 `auth/professor` 축과 교수 캐릭터 에셋 생성 흐름을, 개발자 B는
  `study/dialogue/result` 축을 우선 담당하고, 공통 코드는 application interface와
  공유 계약 문서에서만 맞춘다.
- **근거**: 로그인/교수 준비와 공부 루프 정산은 의존은 있지만 파일 충돌 없이
  병렬 진행이 가능하다.
- **검토한 대안**:
  - API 레이어/DB 레이어 기준 분할: 동일 도메인을 여러 층에서 동시에 건드리게 되어
    충돌 가능성이 커짐
