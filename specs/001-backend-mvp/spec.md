# 기능 명세: 백엔드 MVP 기반

**기능 ID**: `001-backend-mvp`  
**작업 브랜치**: `main`  
**작성일**: 2026-04-18  
**상태**: 초안  
**입력**: 사용자 설명: "BE만을 기준으로 진행해. FE는 따로 진행할거야. Docs/ 안의 파일들을 참고해."

## 사용자 시나리오 및 테스트 *(필수)*

이 스펙은 서비스 전체 MVP 중 백엔드 범위만 정의한다. 사용자 시나리오는 최종 사용자
가치를 기준으로 서술하되, 프론트엔드는 별도 트랙에서 같은 계약 문서를 소비한다고
가정한다.

### 사용자 스토리 1 - 계정 및 교수 데이터 준비 (우선순위: P1)

사용자는 서비스 이용을 시작하기 위해 계정을 만들고 시험 종료일을 저장한 뒤, 공략할
교수 정보를 등록하고, 교수 원본 사진 기반 캐릭터 에셋 상태를 다시 조회할 수 있어야 한다.

**우선순위 근거**: 공부 세션, 호감도, 이벤트, 엔딩은 모두 사용자 계정과 교수
데이터가 먼저 준비되어야 의미가 있다.

**독립 검증 방법**: 신규 사용자가 계정을 생성하고 교수 한 명 이상을 등록한 뒤, 같은
계정으로 로그인하여 저장된 교수 목록, 기본 호감도 상태, 캐릭터 에셋 생성 상태 또는
기본 에셋 대체 상태를 다시 확인할 수 있으면 독립적으로 검증된다.

**수용 시나리오**:

1. **조건** 신규 사용자가 아직 계정을 만들지 않았고, **행동** 이름, 로그인 정보,
   비밀번호, 시험 종료일을 제출하면, **결과** 서비스는 계정을 저장하고 이후 기능에서
   사용할 식별 가능한 사용자 상태를 반환한다.
2. **조건** 인증된 사용자가 교수 정보와 교수 성별을 제출했고, **행동** 등록을
   완료하면, **결과** 서비스는 `male` 또는 `female`만 허용하고 교수명, 성격,
   캐릭터 에셋 상태를 포함한 교수 정보를 저장해 이후 조회에서 동일한 데이터를
   반환한다.
3. **조건** 인증된 사용자가 교수 원본 사진을 함께 등록했고, **행동** 등록 직후
   상세 조회를 수행하면, **결과** 서비스는 `nanobanana` 기반 캐릭터 에셋 생성 상태와
   대표 에셋 정보 또는 기본 에셋 대체 상태를 반환한다.

---

### 사용자 스토리 2 - 공부 세션 정산 및 이벤트 처리 (우선순위: P1)

사용자는 공부를 시작하고 종료했을 때 유효 공부 시간, 호감도 변화, 방치 패널티,
기본 반응 대사, 시나리오 이벤트 여부를 일관된 규칙으로 받아야 한다.

**우선순위 근거**: 이 제품의 핵심 루프는 공부 세션 종료 시점의 감정 보상이다.
세션 정산 규칙이 없으면 서비스의 본질적인 가치가 동작하지 않는다.

**독립 검증 방법**: 기존 사용자와 교수를 준비한 상태에서 공부 세션 시작, 이탈 기록,
공부 종료, 이벤트 선택까지 수행했을 때 문서화된 정산 규칙과 동일한 결과가 나오면
독립적으로 검증된다.

**수용 시나리오**:

1. **조건** 사용자가 유효 공부 시간을 충분히 채운 공부 세션을 종료했고,
   **행동** 서비스가 세션을 정산하면, **결과** 유효 공부 시간에서 이탈 시간이 제외된
   결과와 호감도 증가량, 종료 대사를 함께 반환한다.
2. **조건** 사용자가 공부 중 이탈했다가 2분 이상 뒤에 복귀했고, **행동** 세션을
   종료하면, **결과** 서비스는 해당 이탈 구간에 대해 1회 패널티를 적용하고 이를
   결과 요약에 반영한다.
3. **조건** 사용자가 공부 종료 후 이벤트 대상이 되었고, **행동** 선택지를 고르면,
   **결과** 서비스는 선택 결과 대사와 최종 호감도 변화량을 한 번만 확정한다.

---

### 사용자 스토리 3 - 최종 결과 산출 (우선순위: P2)

사용자는 시험 종료일이 지난 뒤 최종 시험 점수를 제출하고, 교수 관계 결과가 반영된
최종 결과를 받아야 한다.

**우선순위 근거**: 엔딩은 핵심 루프의 마무리이지만, 서비스의 일상 사용은 공부
세션 처리만으로도 성립하므로 P2로 둔다.

**독립 검증 방법**: 시험 종료일이 지난 사용자 계정에서 최종 점수를 제출했을 때,
대표 교수, 최고/최저 호감도 교수, 점수 구간 반응이 함께 반환되면 독립적으로
검증된다.

**수용 시나리오**:

1. **조건** 시험 종료일이 지난 사용자가 있고, **행동** 최종 시험 점수를 제출하면,
   **결과** 서비스는 대표 엔딩 교수와 최고/최저 호감도 교수의 결과 반응을 함께
   반환한다.

## 예외 및 경계 상황

- 사용자가 이미 활성 공부 세션을 가진 상태에서 새 세션을 시작하면 어떻게 처리되는가?
- 사용자가 교수 등록 시 `male`, `female` 이외의 값을 제출하면 어떻게 처리되는가?
- 사용자가 교수 원본 사진 없이 교수를 등록하거나, `nanobanana` 기반 캐릭터 에셋
  생성이 실패하면 어떤 기본 에셋 상태를 반환하는가?
- 사용자가 숨김 상태에서 돌아오지 않은 채 세션 종료를 시도하면 누락된 이탈 구간은
  어떻게 마감되는가?
- 프론트엔드가 별도로 진행되는 동안 계약 문서보다 앞선 필드나 규칙을 기대하면
  어떻게 차이를 추적하고 차단하는가?

## 프론트엔드 / 백엔드 경계 *(필수)*

### 프론트엔드 책임

- 사용자 입력 수집, 화면 상태 표시, 타이머 시각화, 가시성 이벤트 감지는 프론트엔드
  별도 트랙의 책임이다.
- 프론트엔드는 `Docs/`에 기록된 계약 문서 범위 안에서만 백엔드와 통신한다.
- 프론트엔드 화면 구조, 문구, 로컬 상태 관리 방식은 이 스펙 범위 밖이다.

### 백엔드 책임

- 사용자 계정, 시험 종료일, 교수 정보, 호감도 상태를 저장하고 조회한다.
- 교수 원본 사진 참조를 저장하고 `nanobanana` 기반 캐릭터 에셋 생성과 대체 기본
  에셋 규칙을 관리한다.
- 공부 세션 생성, 활성 세션 제약, 이탈 구간 누적, 공부 종료 정산을 처리한다.
- 대사 후보 선택, 이벤트 발생 여부, 이벤트 선택 결과, 최종 결과 산출을 책임진다.
- 프론트엔드와 독립적으로 검증 가능한 계약 응답을 제공한다.

### 공유 계약

- 버전이 명시된 기준 계약 문서는 `Docs/api-spec-v0.1.md`다.
- 도메인 규칙 기준은 `Docs/developer-spec-v0.1.md`와 `Docs/product-plan-v1.0.md`다.
- 대사 및 시나리오 렌더 계약 기준은 `Docs/dialogue-content-spec-v0.1.md`다.
- 시나리오 원문 기준은 `Backend/Reference/` 아래 마크다운 문서다.
- 계약 변경이나 작업 중 질문은 `Docs/FE-comment-v1.0.md`와
  `Docs/BE-comment-v1.0.md`에 기록한다.

### 병렬 작업 계획

- 백엔드는 현재 스펙과 계약 문서를 기준으로 계정, 교수, 세션, 이벤트, 결과 흐름을
  독립적으로 구현할 수 있다.
- 프론트엔드는 별도 진행하되, 새로운 요구나 계약 차이는 코멘트 문서에 먼저 기록한 뒤
  계약 문서를 갱신해야 한다.
- 문서화된 계약이 유지되는 한 FE와 BE는 동일 기능 범위 안에서도 병렬 작업이 가능하다.

## 요구사항 *(필수)*

### 기능 요구사항

- **FR-001**: 시스템은 신규 사용자가 이름, 로그인 정보, 비밀번호, 시험 종료일을
  제출하면 계정을 생성하고 이후 식별 가능한 사용자 상태를 반환해야 한다.
- **FR-002**: 시스템은 인증된 사용자가 교수 정보를 등록하고 조회할 수 있게 해야 하며,
  교수 등록 시 `gender`는 `male`, `female`만 허용하고 교수별 호감도 상태를
  유지해야 한다.
- **FR-003**: 시스템은 교수 등록 시 원본 사진이 있으면 이를 저장하고 `nanobanana`
  기반 여러 표정/포즈의 전신 2D 캐릭터 에셋 생성 절차를 시작해야 하며, 원본 사진이
  없거나 생성에 실패하면 기본 캐릭터 에셋 세트를 사용해야 한다.
- **FR-004**: 시스템은 사용자당 동시에 하나의 활성 공부 세션만 허용해야 한다.
- **FR-005**: 시스템은 공부 세션 시작, 이탈 구간 기록, 공부 종료를 받아 유효 공부
  시간을 계산해야 하며, 숨김 상태 시간은 공부 시간에서 제외해야 한다.
- **FR-006**: 시스템은 유효 공부 시간에 따라 공부 보상을 계산해 등록된 모든 교수의
  호감도에 동일하게 반영하고, 중도 포기 및 방치 규칙에 따른 패널티는 현재 교수 기준
  최종 호감도 변화에 반영해야 한다.
- **FR-007**: 시스템은 공부 종료 시 기본 반응 대사를 반환해야 하며, 조건을 만족하면
  시나리오 이벤트 여부와 선택지를 함께 반환해야 한다.
- **FR-008**: 시스템은 시나리오 이벤트 선택 결과를 한 번만 처리해야 하며, 선택
  결과에 따른 대사와 호감도 변화를 저장해야 한다.
- **FR-009**: 시스템은 시험 종료일 이후에만 최종 시험 점수 제출을 허용해야 하며,
  대표 교수와 최고/최저 호감도 교수 결과를 함께 반환해야 한다.
- **FR-010**: 시스템은 프론트엔드가 별도 진행되더라도 계약 밖의 가정이 생기지 않도록
  버전이 명시된 `Docs/` 문서와 코멘트 문서를 기준 계약으로 유지해야 한다.

### 핵심 엔티티 *(데이터가 포함되는 기능이므로 작성)*

- **사용자 계정**: 로그인 식별자, 사용자명, 시험 종료일을 가진 서비스 이용 주체
- **교수 프로필**: 교수명, 성격, 교수 원본 사진 참조, 캐릭터 에셋 상태를 가진
  공략 대상 정보
- **교수 캐릭터 에셋**: 여러 표정과 포즈의 전신 2D 게임용 이미지를 교수별로 관리하는
  자산 정보
- **호감도 상태**: 사용자-교수 조합마다 유지되되, 공부 보상은 모든 교수에 동일하게
  반영되는 관계 수치
- **공부 세션**: 시작 시각, 종료 시각, 이탈 구간, 유효 공부 시간, 정산 결과를 담는 기록
- **시나리오 이벤트**: 공부 종료 후 발생 가능한 대사/선택지/결과 묶음
- **최종 결과**: 시험 점수와 교수별 관계 상태를 바탕으로 생성되는 엔딩 요약

## 문서 및 커뮤니케이션 영향 *(필수)*

- **생성/갱신 문서**: `Docs/api-spec-v0.1.md`, `Docs/developer-spec-v0.1.md`,
  `Docs/dialogue-content-spec-v0.1.md`, `Docs/BE-comment-v1.0.md`,
  `Backend/Reference/` 관련 시나리오 원문
- **API / 계약 변경점**: 현재 `Docs/api-spec-v0.1.md`의 인증, 교수, 공부 세션,
  이벤트, 최종 결과 계약을 백엔드 구현 기준으로 고정한다. 특히 교수 등록/조회 계약은
  `male`, `female` 성별 제한과 `sourcePhotoUrl`, `characterAssetStatus`,
  `representativeAssetUrl`, `characterAssets` 필드를 포함해야 한다. 구현 중 계약이
  바뀌면 문서를 먼저 갱신해야 한다.
- **대사 콘텐츠 동기화 규칙**: 시나리오 구조, 호감도 구간, 스크립트 라인 포맷,
  플레이스홀더 치환 규칙이 바뀌면 `Docs/dialogue-content-spec-v0.1.md`와
  `Backend/Reference/` 원문을 같은 변경 세트에서 함께 갱신해야 한다.
- **프론트엔드 미해결 질문**: 없음. 프론트엔드에서 새로운 요구가 생기면
  `Docs/FE-comment-v1.0.md`에 기록한다.
- **백엔드 미해결 질문**: 없음. 구현 중 세부 판단 차이는 `Docs/BE-comment-v1.0.md`에
  기록한다.
- **버전 관리 계획**: 계약 필드 추가나 응답 해석 확장은 MINOR, 하위 호환을 깨는
  응답 변경은 MAJOR, 설명 보강은 PATCH로 문서 버전을 올린다.

## 성공 기준 *(필수)*

### 측정 가능한 결과

- **SC-001**: 신규 사용자는 한 번의 가입 흐름 안에서 계정을 만들고 최소 1명의 교수
  정보를 저장한 뒤 즉시 같은 데이터와 캐릭터 에셋 상태를 다시 조회할 수 있어야 한다.
- **SC-002**: 사용자는 유효한 공부 종료 요청 후 95% 이상에서 2초 이내에 공부 시간,
  호감도 변화, 종료 반응을 포함한 결과를 받아야 한다.
- **SC-003**: 정상 공부, 중도 포기, 방치 발생, 이벤트 선택, 최종 결과 제출을 포함한
  핵심 비즈니스 규칙 검증 시나리오에서 문서화된 결과 일치율이 100%여야 한다.
- **SC-004**: 프론트엔드 별도 트랙은 백엔드 구현 기간 동안 `Docs/`에 기록되지 않은
  필드나 규칙 없이 계약 문서만으로 통합 준비를 진행할 수 있어야 한다.

## 가정

- MVP에서는 일반 사용자 한 종류만 존재하며 별도 관리자 기능은 다루지 않는다.
- 프론트엔드는 사용자 입력과 가시성 이벤트를 수집해 백엔드에 전달하지만, 화면 세부
  동작은 별도 트랙에서 정의한다.
- 교수 원본 사진 업로드 자체는 별도 스토리지 흐름에서 완료되며, 백엔드 MVP 계약은
  업로드가 끝난 `sourcePhotoUrl`을 입력으로 받는다고 가정한다.
- 교수별 대사와 이벤트 콘텐츠는 백엔드가 조회 가능한 상태로 준비되어 있거나 시드
  데이터로 제공된다고 가정한다.
- 문서에 정의된 세션 정산 규칙과 결과 산출 규칙이 이번 백엔드 MVP의 기준 범위다.
